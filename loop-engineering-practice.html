Loop Engineering 实践介绍

Loop Engineering 是一种把 AI 编程从“一次性问答”升级为“目标驱动、证据闭环、持续沉淀”的工程实践。 它要求每一次需求开发都能形成清晰目标、可追踪执行、可验证结果和可复用经验。

Chapter 01

Loop Engineering 是什么

它把需求开发拆成可反复运行的工程循环：目标、上下文、计划、执行、验证、沉淀。

Chapter 02

怎么利用 Codex /goal 实现 Loop Engineering

/goal 的价值是把一次对话升级为一个持续追踪的工程目标。目标完成前，Codex 会围绕它持续推进、验证并汇报状态。

G

用 /goal 建立目标容器

把业务目标、交付物、验收条件写进目标描述，让每轮对话都继承清晰背景。

C

让 Codex 先补齐上下文

要求它读取源码、文档、配置和运行日志，再输出判断，并明确标注证据状态和必要假设。

D

执行设计、实现、部署

Codex 可以在本地修改文件、运行命令、远程 SSH 部署，过程以短状态更新保持可追踪。

V

用证据完成目标

最终输出应包含改了什么、如何验证、哪些风险还存在，并在目标完成后关闭循环。

/goal 基于新版 UI 源码完成某业务模块需求开发： 1. 先从页面、接口、状态管理和路由中总结系统需求； 2. 给出最小设计方案和影响范围； 3. 修改代码并补充必要测试； 4. 通过 SSH 部署到 CA 测试服务器； 5. 用浏览器 E2E 验证关键流程； 6. 把经验、风险和后续注意事项写回仓库内 md 业务记忆。

Chapter 03

CA 项目的当前实践路径

CA 项目已经可以把“理解新版本 UI 源码、总结系统需求、设计并实现、远程部署、浏览器 E2E 验证”串成一条需求开发流水线。

源码理解 读取新版 UI 页面、组件、路由、状态、接口调用和权限逻辑。
需求总结 把源码行为整理为系统需求、边界、异常状态和验收条件。
设计实现 形成最小设计，完成代码修改、配置调整和必要测试。
远程部署 通过 SSH 进入服务器，执行构建、发布、重启和日志检查。
浏览器 E2E 打开真实环境页面，模拟用户路径，保存截图、日志和结论。

当前可形成的交付物

  • ✓系统需求表：页面行为、接口依赖、权限规则、异常分支、验收标准。
  • ✓设计说明：影响范围、数据流、模块改动、兼容策略、回滚方式。
  • ✓实现变更：代码 diff、测试补充、配置调整和部署脚本记录。
  • ✓验证证据：浏览器 E2E 路径、截图、网络请求、控制台日志、服务端日志。
  • ✓业务记忆：通过仓库内 md 文档记录问题原因、修复路径、项目约定和下一次可复用的提示词。

一条需求的推荐闭环

  • ✓从新 UI 源码出发，同时对照原型，让需求理解和实际实现保持一致。
  • ✓先形成需求摘要和设计摘要，再进入编码，便于人工快速校正方向。
  • ✓部署动作通过 SSH 留痕，重点记录命令、环境、版本和日志异常。
  • ✓E2E 验证以真实浏览器路径为准，覆盖主流程、失败态和权限态。
  • ✓收尾时更新仓库内 md 业务记忆，让项目经验进入下一轮上下文。

关键原则：AI 要交付“需求如何得出、设计为何这样、部署是否成功、用户路径是否可用”的完整证据链。

Chapter 04

未来计划：从需求开发到自动修复缺陷

下一步可以把 Loop Engineering 扩展成面向进度看板和缺陷卡片的自动化平台能力，让 AI 围绕真实项目流持续工作。

接入 ONES 进度看板

以需求卡片、缺陷卡片、优先级、负责人、环境信息和验收标准作为目标输入，自动创建 AI 工作循环。

从缺陷卡片自动修复

AI 读取缺陷描述、复现步骤、截图、日志和关联版本，自动复现、定位、设计修复、提交补丁并执行验证。

建设 CA QA 机器人

在仓库内 md 业务记忆稳定沉淀后，把问题现象、根因、修复方式、验证路径、环境差异和项目约定接入 QA 机器人知识库。

基于 GitLab 建立人机协作控制点

使用 GitLab issue 和 PR 承载需求确认、修复方案、代码评审、验证证据和发布确认，让人机协作过程可追踪、可审查、可复用。

平台输入

ONES 卡片、代码仓库、部署环境、测试账号、日志入口、仓库内 md 业务记忆。

平台处理

目标拆解、上下文收集、自动修复、远程部署、E2E 验证、风险报告。

平台输出

修复结果、验证证据、变更说明、业务记忆更新、下一步待办。

Loop Engineering = 目标驱动 + 证据闭环 + 经验沉淀

对 CA 项目而言，它的核心价值在于把需求分析、实现、部署、验证和知识维护串成可复用的 AI 工程循环。
